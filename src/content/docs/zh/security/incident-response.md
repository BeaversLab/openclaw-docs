## 1. 检测和分类

我们监控来自以下来源的安全信号：

- GitHub 安全公告 (GHSA) 和私有漏洞报告。
- 当报告不敏感时，使用公开的 GitHub 议题/讨论。
- 自动信号（例如 Dependabot、CodeQL、npm 公告和机密扫描）。

初步分类：

1. 确认受影响的组件、版本以及信任边界的影响。
2. 使用仓库 `SECURITY.md` 范围和范围外规则，将其分类为安全问题、加固项或无需处理项。
3. 事件负责人会做出相应的响应。

## 2. 评估

严重程度指南：

- **严重：** 软件包/版本/仓库遭到入侵、正在被利用，或未经验证的信任边界绕过导致高影响的控制或数据泄露。
- **高：** 经验证的信任边界绕过，需要有限的先决条件（例如已认证但未授权的高影响操作），或泄露 OpenClaw 拥有的敏感凭据。
- **中：** 具有实际影响但利用受限或需要大量先决条件的重大安全弱点。
- **低：** 纵深防御发现、范围狭窄的拒绝服务，或未显示信任边界绕过的加固/一致性差距。

## 3. 响应

1. 向报告者确认收到（若内容敏感则私下回复）。
2. 在受支持的版本和最新的 `main` 上复现，然后实施并验证补丁，并提供回归覆盖。
3. 对于严重/高级别事件，尽可能快地准备修补版本。
4. 对于中/低级别事件，在正常发布流程中打补丁，并记录缓解指导。

## 4. 沟通

我们通过以下方式进行沟通：

- 受影响仓库中的 GitHub 安全公告。
- 已修复版本的发布说明/更新日志条目。
- 直接向报告者跟进状态和解决方案。

披露政策：

- 严重/高级别的事件应进行协调披露，并在适当的情况下发布 CVE。
- 根据影响范围和用户受影响情况，低风险的加固发现可能会记录在发行说明或公告中，而不发布 CVE。

## 5. 恢复与后续跟进

发布修复程序后：

1. 在 CI 和发布产物中验证补救措施。
2. 进行简短的事后复盘（时间线、根本原因、检测缺口、预防计划）。
3. 添加后续的加固/测试/文档任务，并跟踪直至完成。
