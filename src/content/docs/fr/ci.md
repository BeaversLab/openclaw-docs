# Pipeline CI

La CI s'exécute à chaque push vers `main` et chaque pull request. Elle utilise un ciblage intelligent pour ignorer les tâches coûteuses lorsque seules des zones non liées ont changé.

QA Lab dispose de voies CI dédiées en dehors du workflow principal à portée intelligente. Le workflow `Parity gate` s'exécute sur les modifications de PR correspondantes et le déclenchement manuel ; il construit l'environnement d'exécution QA privé et compare les packs d'agence GPT-5.4 et Opus 4.6 simulés. Le workflow `QA-Lab - All Lanes` s'exécute nightly sur `main` et sur déclenchement manuel ; il répartit la porte de parité simulée, la voie Matrix en direct, et la voie Telegram en direct en tant que travaux parallèles. Les travaux en direct utilisent l'environnement `qa-live-shared`, et la voie Telegram utilise des baux Convex. `OpenClaw Release Checks` exécute également les mêmes voies QA Lab avant l'approbation de la release.

## Fail-Fast Order

Les tâches sont ordonnées de sorte que les vérifications peu coûteuses échouent avant l'exécution des plus coûteuses :

1. `preflight` décide quelles voies existent. La logique `docs-scope` et `changed-scope` sont des étapes à l'intérieur de cette tâche, et non des tâches autonomes.
2. `security-scm-fast`, `security-dependency-audit`, `security-fast`, `check`, `check-additional`, `check-docs`, et `skills-python` échouent rapidement sans attendre les tâches plus lourdes d'artefacts et de matrice de plateformes.
3. `build-artifacts` chevauche les voies Linux rapides afin que les consommateurs en aval puissent démarrer dès que la construction partagée est prête.
4. Ensuite, les voies de plateformes et d'exécution plus lourdes se déploient : `checks-fast-core`, `checks-fast-contracts-channels`, `checks-node-extensions`, `checks-node-core-test`, `extension-fast` uniquement pour PR, `checks`, `checks-windows`, `macos-node`, `macos-swift`, et `android`.

La logique de portée réside dans `scripts/ci-changed-scope.mjs` et est couverte par des tests unitaires dans `src/scripts/ci-changed-scope.test.ts`.
Les modifications du workflow CI valident le graphe CI Node ainsi que le linting du workflow, mais ne forcent pas les builds natifs Windows, Android ou macOS par elles-mêmes ; ces volets de plateforme restent limités aux modifications des sources de la plateforme.
Les vérifications Node Windows sont limitées aux wrappers de processus/chemins spécifiques à Windows, aux assistants d'exécution npm/pnpm/UI, à la configuration du gestionnaire de packages et aux surfaces du workflow CI qui exécutent cette voie ; les modifications non liées aux sources, aux plugins, aux install-smoke et aux test-only restent sur les volets Node Linux afin qu'elles ne réservent pas un worker Windows à 16 vCPU pour une couverture qui est déjà exercée par les fragments de test normaux.
Le workflow séparé `install-smoke` réutilise le même script de portée via son propre travail `preflight`. Il calcule `run_install_smoke` à partir du signal de changed-smoke plus étroit, de sorte que le smoke Docker/install s'exécute pour les modifications d'installation, d'empaquetage, liées aux conteneurs, les modifications de production d'extension groupée, et les surfaces principales du plugin/channel/gateway/Plugin SDK que les jobs smoke Docker exercent. Les modifications test-only et docs-only ne réservent pas de workers Docker. Son smoke de package QR force la couche Docker `pnpm install` à se réexécuter tout en préservant le cache de magasin pnpm de BuildKit, exerçant ainsi l'installation sans retélécharger les dépendances à chaque exécution. Son e2e gateway-network réutilise l'image d'exécution construite plus tôt dans le travail, ajoutant ainsi une couverture WebSocket réel de conteneur à conteneur sans ajouter une autre construction Docker. Le `test:docker:all` local préconstruit une image d'application construite `scripts/e2e/Dockerfile` partagée et la réutilise sur les exécuteurs de smoke de conteneur E2E ; le workflow live/E2E réutilisable miroir ce modèle en construisant et en poussant une image E2E Docker GHCR étiquetée SHA avant la matrice Docker, puis en exécutant la matrice avec `OPENCLAW_SKIP_DOCKER_BUILD=1`. Les tests QR et Docker d'installateur conservent leurs propres Dockerfiles axés sur l'installation. Un travail séparé `docker-e2e-fast` exécute le profil Docker bundled-plugin borné sous un délai d'expiration de commande de 120 secondes : réparation des dépendances setup-entry plus isolement des échecs du bundled-loader synthétique. La matrice complète de mise à jour/groupée de canal reste manuelle/suite complète car elle effectue des passes répétées de mise à jour npm réelle et de réparation doctor.

La logique locale de voie modifiée réside dans `scripts/changed-lanes.mjs` et est exécutée par `scripts/check-changed.mjs`. Cette porte locale est plus stricte quant aux limites de l'architecture que la portée générale de la plateforme CI : les modifications de production du cœur exécutent la vérification de type de production du cœur ainsi que les tests du cœur, les modifications de test uniquement du cœur n'exécutent que la vérification de type/tests de test du cœur, les modifications de production des extensions exécutent la vérification de type de production des extensions ainsi que les tests des extensions, et les modifications de test uniquement des extensions n'exécutent que la vérification de type/tests de test des extensions. Les modifications du SDK Public Plugin ou du plugin-contract s'étendent à la validation des extensions car les extensions dépendent de ces contrats fondamentaux. Les mises à jour de version de métadonnées de publication uniquement exécutent des vérifications ciblées de version/config/root-dependency. Les modifications inconnues de racine/config échouent en mode sans échec sur toutes les voies.

Lors des poussées (pushes), la matrice `checks` ajoute la voie `compat-node22` réservée aux poussées. Sur les demandes de tirage (pull requests), cette voie est ignorée et la matrice reste concentrée sur les voies de test/channel normales.

Les familles de tests Node les plus lentes sont divisées ou équilibrées afin que chaque travail reste petit : les contrats de canal divisent la couverture du registre et du cœur en six fragments pondérés au total, les tests de plugins groupés s'équilibrent sur six workers d'extension, la réponse automatique s'exécute sous forme de trois workers équilibrés au lieu de six petits workers, et les configurations de passerelle/plugin agentiques sont réparties sur les travaux Node agentiques existants sources uniquement au lieu d'attendre les artefacts construits. Les tests étendus de navigateur, QA, multimédia et divers plugins utilisent leurs configurations Vitest dédiées au lieu du fichier de rattrapage de plugin partagé. La voie des agents étendus utilise l'ordonnanceur parallèle de fichiers Vitest partagé car il est dominé par l'importation/la planification plutôt que par un seul fichier de test lent. `runtime-config` s'exécute avec le fragment de runtime d'infra central pour empêcher le fragment de runtime partagé de posséder la queue. `check-additional` maintient le travail de compilation/canary des limites de paquets ensemble et sépare l'architecture de la topologie d'exécution de la couverture de surveillance de la passerelle ; le fragment de garde de frontière exécute ses petites gardes indépendantes simultanément dans un seul travail. La surveillance de passerelle, les tests de canal et le fragment de support central aux limites s'exécutent simultanément dans `build-artifacts` une fois que `dist/` et `dist-runtime/` sont déjà construits, en conservant leurs anciens noms de vérification en tant que travaux de vérification légers tout en évitant deux workers Blacksmith supplémentaires et une seconde file d'attente de consommateur d'artefacts.
Le CI Android exécute à la fois `testPlayDebugUnitTest` et `testThirdPartyDebugUnitTest`, puis construit l'APK de débogage Play. La variante tierce n'a pas de jeu de sources ou de manifeste distinct ; sa voie de tests unitaires compile toujours cette variante avec les indicateurs BuildConfig SMS/journal d'appels, tout en évitant un travail de packaging APK de débogage en double à chaque poussée pertinente pour Android.
`extension-fast` est réservé aux PR car les exécutions de poussée exécutent déjà tous les fragments de plugins groupés. Cela permet de conserver des commentaires sur les plugins modifiés pour les révisions sans réserver un worker Blacksmith supplémentaire sur `main` pour une couverture déjà présente dans `checks-node-extensions`.

GitHub peut marquer les travaux remplacés comme `cancelled` lorsqu'un nouvel envoi (push) atterrit sur la même PR ou la référence `main`. Considérez cela comme du bruit CI, sauf si l'exécution la plus récente pour la même référence échoue également. Les vérifications de fragments (shards) agrégés utilisent `!cancelled() && always()` afin qu'ils signalent toujours les échecs de fragments normaux, mais ne se mettent pas en file d'attente une fois que l'ensemble du workflow a déjà été remplacé.
La clé de concurrence CI est versionnée (`CI-v7-*`), de sorte qu'un zombie côté GitHub dans un ancien groupe de file d'attente ne peut pas bloquer indéfiniment les nouvelles exécutions sur main.

## Équivalents locaux

```bash
pnpm changed:lanes   # inspect the local changed-lane classifier for origin/main...HEAD
pnpm check:changed   # smart local gate: changed typecheck/lint/tests by boundary lane
pnpm check          # fast local gate: production tsgo + sharded lint + parallel fast guards
pnpm check:test-types
pnpm check:timed    # same gate with per-stage timings
pnpm build:strict-smoke
pnpm check:architecture
pnpm test:gateway:watch-regression
pnpm test           # vitest tests
pnpm test:channels
pnpm test:contracts:channels
pnpm check:docs     # docs format + lint + broken links
pnpm build          # build dist when CI artifact/build-smoke lanes matter
node scripts/ci-run-timings.mjs <run-id>  # summarize wall time, queue time, and slowest jobs
```
