# Unified Runtime Streaming 重構計畫

## 目標

為 `main`、`subagent` 和 `acp` 提供一個共用的串流管線，以便所有執行時期都獲得相同的合併、分塊、遞送順序和崩潰恢復行為。

## 為何存在

- 目前的行為分散在多個特定於執行時的塑形路徑中。
- 格式化/合併錯誤可以在一個路徑中修復，但在其他路徑中仍然存在。
- 交付一致性、重複抑制和恢復語義更難以推斷。

## 目標架構

單一管道、特定於執行時的適配器：

1. 執行時適配器僅發出規範事件。
2. 共享流組裝器合併並定稿文字/工具/狀態事件。
3. 共享通道投射器一次套用特定通道的區塊劃分/格式化。
4. 共享交付分類帳強制執行等冪的傳送/重播語意。
5. 輸出通道配接器執行傳送並記錄交付檢查點。

標準事件合約：

- `turn_started`
- `text_delta`
- `block_final`
- `tool_started`
- `tool_finished`
- `status`
- `turn_completed`
- `turn_failed`
- `turn_cancelled`

## 工作流

### 1) 標準串流契約

- 在核心中定義嚴格的事件架構 + 驗證。
- 新增配接器契約測試，以保證每個執行階段都發出相容的事件。
- 及早拒絕格式錯誤的執行階段事件，並顯示結構化診斷資訊。

### 2) 共用串流處理器

- 用一個處理器替換特定於運行時的合併器/投影器邏輯。
- 處理器負責文本增量緩衝、空閒刷新、最大塊拆分和完成刷新。
- 將 ACP/main/subagent 配置解析移至一個輔助函數中，以防止漂移。

### 3) 共享通道投影

- 保持通道適配器簡單：接受最終區塊並發送。
- 將 Discord 特定的分塊怪癖僅移至通道投影器。
- 在投影之前保持管線與通道無關。

### 4) 傳遞分類帳 + 重播

- 新增每回合/每區塊的傳遞 ID。
- 在實體傳送之前和之後記錄檢查點。
- 重新啟動時，等幂地重播待處理區塊並避免重複。

### 5) 遷移和切換

- 階段 1：影子模式（新管線計算輸出，但由舊路徑發送；進行比較）。
- 第二階段：各個執行時期逐一切換 (`acp`，然後 `subagent`，然後 `main`，或根據風險反序進行)。
- 第三階段：刪除舊版執行時期特定的串流代碼。

## 非目標

- 在此重構中不變更 ACP 原則/許可權模型。
- 除投影相容性修復外，不進行特定通道的功能擴充。
- 無傳輸/後端重新設計（除非需要事件同位，否則 acpx 外掛程式合約保持不變）。

## 風險與緩解措施

- 風險：現有主/代理路徑中的行為回歸。
  緩解措施：陰影模式差異比較 + 介面卡合約測試 + 通道端到端測試。
- 風險：當機恢復期間的重複發送。
  緩解措施：持久傳遞 ID + 傳遞介面器中的等幂重播。
- 風險：執行時適配器再次分岔。
  緩解措施：所有適配器都需要共享的合約測試套件。

## 驗收標準

- 所有執行時都通過共享的串流合約測試。
- Discord ACP/main/subagent 針對微小的增量產生等效的間距/分塊行為。
- 當機/重新啟動重播不會針對相同的傳遞 ID 發送重複的分塊。
- 舊版 ACP 投影器/合併器路徑已移除。
- 串流設定解析是共享的且獨立於執行時。
