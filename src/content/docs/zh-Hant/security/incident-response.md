## 1. 偵測與分類

我們監控來自以下來源的安全訊號：

- GitHub 安全公告 (GHSA) 與私人的漏洞報告。
- 當報告不具敏感性時，透過公開的 GitHub Issues/Discussions。
- 自動化訊號（例如 Dependabot、CodeQL、npm 公告和秘密掃描）。

初步分類：

1. 確認受影響的元件、版本以及信任邊界的影響。
2. 根據儲存庫 `SECURITY.md` 的範圍 (scope) 與非範圍 (out-of-scope) 規則，將其分類為安全問題、強化措施 (hardening) 或無需採取行動。
3. 事件負責人會進行相應的回應。

## 2. 評估

嚴重性指南：

- **嚴重：** 套件/發行版本/儲存庫遭入侵、正遭受利用，或未經授權的信任邊界繞過，並導致高影響的權限控制或資料外洩。
- **高：** 已確認的信任邊界繞過，且僅需有限的先決條件（例如已認證但未授權的高影響操作），或 OpenClaw 擁有的敏感憑證外洩。
- **中：** 具有實際影響的重大安全弱點，但利用受限或有實質的先決條件。
- **低：** 縱深防禦 發現、範圍狹窄的拒絕服務，或未顯示信任邊界繞過的強化/一致性差異。

## 3. 回應

1. 向報告者確認已收到（若內容敏感則私下回覆）。
2. 在支援的發行版本與最新的 `main` 上重現問題，然後實作並驗證修補程式，並包含回歸測試。
3. 對於嚴重/高級別事件，盡速準備已修補的發行版本。
4. 對於中/低級別事件，在正常的發行流程中進行修補，並記載緩解指引。

## 4. 溝通

我們透過以下方式進行溝通：

- 受影響儲存庫中的 GitHub 安全公告。
- 已修復版本的發行說明/變更日誌條目。
- 直接向報告者回報狀態與解決方案。

揭露政策：

- 嚴重/高級別事件應接受協調揭露，並視情況發布 CVE。
- 低風險的強化發現可能會記載於發行說明或公告中而不發布 CVE，取決於其影響與使用者暴露程度。

## 5. 復原與後續追蹤

在發布修補程式之後：

1. 在 CI 和發布產物中驗證補救措施。
2. 進行簡短的事故後檢討（時間線、根本原因、檢測缺口、預防計畫）。
3. 新增後續的加固、測試或文件任務，並追蹤其至完成狀態。
