本頁面記錄了 2026 年 5 月 OpenClaw 效能、套件大小、相依性和 shrinkwrap 清理背後的證據。這是公開部落格文章的技術伴侶文件。

這裡結合了兩項稽核：

- **Release performance sweep:** 從 `v2026.5.28` 回溯至穩定版 `v2026.4.23` 的 GitHub Releases，使用 `OpenClaw Performance` 工作流程，`profile=smoke`，mock-provider lane。大多數標籤行為單一樣本；`v2026.5.27` 和 `v2026.5.28` 行使用最新的 repeat-3 release-branch 成品。
- **Earlier April context:** 從 `v2026.4.1` 到 `v2026.5.2` 發布了 `clawgrit-reports` mock-provider 基準，僅用於避免將損壞的 4 月下旬版本作為公開效能基準。
- **Install footprint sweep:** 將 `npm install --ignore-scripts` 全新安裝到臨時套件中，使用 `du -sk node_modules` 測量大小，並使用 `node_modules` walk 計算套件實例數量。
- **npm package size sweep:** 針對已發布的版本進行 `npm pack openclaw@<version> --dry-run --json`，記錄壓縮後的 tarball 大小、解壓縮大小和檔案數量。

<Warning>主要的效能掃描每個標籤使用一個樣本，但 `v2026.5.27` 和 `v2026.5.28` 行除外，它們使用最新的 repeat-3 release-branch 成品。Early April context 使用來自 `clawgrit-reports` 的已發布 repeat-3 中位數。將這些數字視為趨勢證據和迴歸搜尋信號，而非發布門檻統計數據。</Warning>

## 關鍵數字

請勿將四月下旬的損壞資料列作為公開的效能基準。
`v2026.4.23` 和 `v2026.4.29` 是有用的迴歸證據，但巨大的
`14x` 樣式的差值主要描述了從糟糕的發布版本線中的恢復。

對於部落格敘事，請使用四月較早發布的基準作為規模參考：

| 指標            | 四月較早的基準 | `v2026.5.28` |                  差異 |
| --------------- | -------------: | -----------: | --------------------: |
| Cold agent turn |        9,819ms |      1,908ms | 降低 80.6%，快 5.1 倍 |
| Warm agent turn |        7,458ms |      1,870ms | 降低 74.9%，快 4.0 倍 |
| Agent 峰值 RSS  |        686.2MB |      581.0MB |            降低 15.3% |

較早的四月份基準是來自已發布
`clawgrit-reports` mock-provider 執行的 `v2026.4.14`。該次執行使用了重複 3 次，失敗僅是因為

## 供應鏈解讀

依賴計數是一個營運安全指標，而不僅僅是安裝大小指標。每個套件都會擴展維護者、tarball、傳遞更新、選用原生二進位檔和操作員必須信任的安裝時間行為的集合。

清理方向為：

- 將繁重和選用的功能保留在預設核心安裝之外
- 讓外掛套件擁有自己的執行時期依賴關係圖
- 避免在 Gateway 啟動期間進行執行時期套件管理員修復
- 在不導致所有平台原生套件實例化的情況下，保留確定性安裝
- 在套件驗收和測量路徑中保持安裝腳本停用
- 在發佈前發現嵌套依賴樹和原生選用依賴項爆增
