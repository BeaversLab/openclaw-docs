Implementa OpenClaw en Render utilizando Infraestructura como Código. El `render.yaml` Blueprint incluido define toda tu pila de forma declarativa: servicio, disco, variables de entorno, para que puedas implementar con un solo clic y versionar tu infraestructura junto con tu código.

## Requisitos previos

- Una [cuenta de Render](https://render.com) (nivel gratuito disponible)
- Una clave de API de tu [proveedor de modelos](/es/providers) preferido

## Implementar con un Render Blueprint

[Implementar en Render](https://render.com/deploy?repo=https://github.com/openclaw/openclaw)

Al hacer clic en este enlace:

1. Creará un nuevo servicio de Render desde el `render.yaml` Blueprint en la raíz de este repositorio.
2. Te pedirá que configures `SETUP_PASSWORD`
3. Construirá la imagen de Docker e implementará

Una vez implementado, la URL de tu servicio sigue el patrón `https://<service-name>.onrender.com`.

## Entendiendo el Blueprint

Los Render Blueprints son archivos YAML que definen tu infraestructura. El `render.yaml` en este
repositorio configura todo lo necesario para ejecutar OpenClaw:

```yaml
services:
  - type: web
    name: openclaw
    runtime: docker
    plan: starter
    healthCheckPath: /health
    envVars:
      - key: PORT
        value: "8080"
      - key: SETUP_PASSWORD
        sync: false # prompts during deploy
      - key: OPENCLAW_STATE_DIR
        value: /data/.openclaw
      - key: OPENCLAW_WORKSPACE_DIR
        value: /data/workspace
      - key: OPENCLAW_GATEWAY_TOKEN
        generateValue: true # auto-generates a secure token
    disk:
      name: openclaw-data
      mountPath: /data
      sizeGB: 1
```

Características clave del Blueprint utilizadas:

| Característica        | Propósito                                                            |
| --------------------- | -------------------------------------------------------------------- |
| `runtime: docker`     | Construye desde el Dockerfile del repositorio                        |
| `healthCheckPath`     | Render monitorea `/health` y reinicia las instancias poco saludables |
| `sync: false`         | Solicita un valor durante la implementación (secretos)               |
| `generateValue: true` | Genera automáticamente un valor criptográficamente seguro            |
| `disk`                | Almacenamiento persistente que sobrevive a las reimplementaciones    |

## Elegir un plan

| Plan      | Apagado                          | Disco         | Mejor para                     |
| --------- | -------------------------------- | ------------- | ------------------------------ |
| Gratuito  | Después de 15 min de inactividad | No disponible | Pruebas, demostraciones        |
| Inicial   | Nunca                            | 1GB+          | Uso personal, equipos pequeños |
| Estándar+ | Nunca                            | 1GB+          | Producción, múltiples canales  |

El Blueprint por defecto es `starter`. Para usar el nivel gratuito, cambia `plan: free` en el
`render.yaml` de tu fork (pero ten en cuenta: sin disco persistente significa que la configuración se restablece en cada implementación).

## Después de la implementación

### Completar la configuración web

1. Navega a `https://<your-service>.onrender.com/setup`
2. Ingrese su `SETUP_PASSWORD`
3. Seleccione un proveedor de modelos y pegue su clave de API
4. Opcionalmente configure los canales de mensajería (Telegram, Discord, Slack)
5. Haga clic en **Run setup**

### Acceder al Control UI

El panel web está disponible en `https://<your-service>.onrender.com/openclaw`.

## Funciones del panel de Render

### Registros

Vea los registros en tiempo real en **Dashboard → your service → Logs**. Filtre por:

- Registros de compilación (creación de imagen Docker)
- Registros de despliegue (inicio del servicio)
- Registros de tiempo de ejecución (salida de la aplicación)

### Acceso a Shell

Para depurar, abra una sesión de shell a través de **Dashboard → your service → Shell**. El disco persistente está montado en `/data`.

### Variables de entorno

Modifique las variables en **Dashboard → your service → Environment**. Los cambios activan un redespliegue automático.

### Auto-despliegue

Si usa el repositorio original de OpenClaw, Render no desplegará automáticamente su OpenClaw. Para actualizarlo, ejecute una sincronización manual de Blueprint desde el panel.

## Dominio personalizado

1. Vaya a **Dashboard → your service → Settings → Custom Domains**
2. Añada su dominio
3. Configure el DNS como se indica (CNAME a `*.onrender.com`)
4. Render aprovisiona un certificado TLS automáticamente

## Escalado

Render admite el escalado horizontal y vertical:

- **Vertical**: Cambie el plan para obtener más CPU/RAM
- **Horizontal**: Aumente el recuento de instancias (plan Estándar y superior)

Para OpenClaw, el escalado vertical suele ser suficiente. El escalado horizontal requiere sesiones pegajosas o gestión de estado externa.

## Copias de seguridad y migración

Exporte su configuración y espacio de trabajo en cualquier momento:

```
https://<your-service>.onrender.com/setup/export
```

Esto descarga una copia de seguridad portátil que puede restaurar en cualquier host OpenClaw.

## Solución de problemas

### El servicio no se iniciará

Compruebe los registros de despliegue en el panel de Render. Problemas comunes:

- Falta `SETUP_PASSWORD` — el Blueprint lo solicita, pero verifique que esté configurado
- Discrepancia de puerto — asegúrese de que `PORT=8080` coincida con el puerto expuesto del Dockerfile

### Arranques en frío lentos (nivel gratuito)

Los servicios de nivel gratuito se apagan después de 15 minutos de inactividad. La primera solicitud después de apagarse tarda unos segundos mientras se inicia el contenedor. Actualice al plan Starter para estar siempre activo.

### Pérdida de datos después del redespliegue

Esto sucede en el nivel gratuito (sin disco persistente). Actualice a un plan de pago, o
exporte regularmente su configuración a través de `/setup/export`.

### Fallos en los controles de salud

Render espera una respuesta 200 de `/health` dentro de 30 segundos. Si las compilaciones tienen éxito pero las implementaciones fallan, es posible que el servicio esté tardando demasiado en iniciarse. Verifique:

- Registros de compilación para buscar errores
- Si el contenedor se ejecuta localmente con `docker build && docker run`
