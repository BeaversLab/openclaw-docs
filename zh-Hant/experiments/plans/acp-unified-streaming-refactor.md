# Unified Runtime Streaming Refactor Plan

## 目標

提供 `main`、`subagent` 和 `acp` 的單一共用串流管線，以便所有執行時都能獲得相同的合併、分塊、傳遞順序和崩潰恢復行為。

## 存在原因

- 目前的行為分散在多個特定於執行時的塑形路徑中。
- 格式化/合併錯誤可以在一條路徑中修復，但在其他路徑中仍然存在。
- 傳遞一致性、重複抑制和恢復語義更難以推斷。

## 目標架構

單一管線、特定於執行時的轉接器：

1. 執行時轉接器僅發出規範事件。
2. 共用的串流組合器合併並完成文字/工具/狀態事件。
3. 共用的頻道投影器套用特定於頻道的分塊/格式化一次。
4. 共用的傳遞帳本執行等幕傳送/重播語義。
5. 出站頻道轉接器執行傳送並記錄傳遞檢查點。

規範事件合約：

- `turn_started`
- `text_delta`
- `block_final`
- `tool_started`
- `tool_finished`
- `status`
- `turn_completed`
- `turn_failed`
- `turn_cancelled`

## 工作流

### 1) 規範串流合約

- 在核心中定義嚴格的事件架構和驗證。
- 增加轉接器合約測試，以保證每個執行時都發出相容的事件。
- 儘早拒絕格式錯誤的執行時事件，並顯示結構化診斷資訊。

### 2) 共用串流處理器

- 用一個處理器替換特定於執行時的合併器/投影器邏輯。
- 處理器負責文字增量緩衝、閒置排清、最大塊拆分和完成排清。
- 將 ACP/main/subagent 配置解析移至一個輔助函式中，以防止發生分歧。

### 3) 共用通道投影

- 保持通道配接器簡單：接受最終區塊並發送。
- 將 Discord 特有的分塊怪癖僅移至通道投影器。
- 在投影之前，保持管線與通道無關。

### 4) 傳遞帳本 + 重放

- 新增每個回合/每個區塊的傳遞 ID。
- 在實體發送之前和之後記錄檢查點。
- 重啟時，等幂地重放待處理區塊並避免重複。

### 5) 遷移與切換

- 第 1 階段：影子模式（新管線計算輸出，但舊路徑進行發送；進行比較）。
- 第 2 階段：依運行時逐一切換（`acp`，然後是 `subagent`，接著是 `main`，或依風險反向順序）。
- 第 3 階段：刪除舊版特定運行時的串流程式碼。

## 非目標

- 在此重構中不變更 ACP 政策/權限模型。
- 除投影相容性修正外，不進行特定於通道的功能擴充。
- 不重新設計傳輸/後端（除非為了事件同等性需要，否則 acpx 外掛程式合約保持不變）。

## 風險與緩解措施

- 風險：現有 main/subagent 路徑的行為回歸。
  緩解措施：影子模式差異比對 + 配接器合約測試 + 通道端對端測試。
- 風險：崩潰恢復期間發生重複發送。
  緩解措施：持久化傳遞 ID + 傳遞配接器中的等幂重放。
- 風險：運行時配接器再次發生分歧。
  緩解措施：所有配接器強制使用共用合約測試套件。

## 驗收標準

- 所有運行時均通過共用串流合約測試。
- Discord ACP/main/subagent 對於微小的差異產生等效的間距/分塊行為。
- 崩潰/重啟重放時，對於相同的傳遞 ID 不會發送重複的區塊。
- 舊版 ACP 投影器/合併器路徑已移除。
- 串流配置解析已共用且獨立於運行時。
