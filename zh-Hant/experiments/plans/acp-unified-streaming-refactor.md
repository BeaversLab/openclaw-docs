# 統一運行時串流重構計畫

## 目標

為 `main`、`subagent` 和 `acp` 提供一個共享的串流管線，以便所有運行時獲得相同的合併、分塊、傳遞順序和崩潰恢復行為。

## 為什麼存在此計畫

- 目前的行為分散在多個特定於運行時的塑型路徑中。
- 格式化/合併錯誤可以在一條路徑中修復，但在其他路徑中仍然存在。
- 傳遞一致性、重複抑制和恢復語義更難以推斷。

## 目標架構

單一管線，特定於運行時的介面卡：

1. 運行時介面卡僅發出規範事件。
2. 共享串流組裝器合併並最終處理文字/工具/狀態事件。
3. 共享通道投影器僅應用一次特定於通道的分塊/格式化。
4. 共享傳遞帳本執行冪等發送/重播語義。
5. 出站通道介面卡執行發送並記錄傳遞檢查點。

規範事件合約：

- `turn_started`
- `text_delta`
- `block_final`
- `tool_started`
- `tool_finished`
- `status`
- `turn_completed`
- `turn_failed`
- `turn_cancelled`

## 工作流

### 1) 規範串流合約

- 在核心中定義嚴格的事件架構 + 驗證。
- 添加介面卡合約測試以保證每個運行時發出相容的事件。
- 及早拒絕格式錯誤的運行時事件並顯示結構化診斷資訊。

### 2) 共享串流處理器

- 用一個處理器替換特定於運行時的合併器/投影器邏輯。
- 處理器負責文字增量緩衝、閒置刷新、最大塊拆分和完成刷新。
- 將 ACP/main/subagent 配置解析移動到一個輔助程式中，以防止漂移。

### 3) 共享通道投影

- 保持通道介面卡簡單：接受最終處理的區塊並發送。
- 將特定於 Discord 的分塊怪癖僅移動到通道投影器。
- 在投射之前，讓管線保持與通道無關。

### 4) 遞送帳本 + 重放

- 新增每個回合/每個區塊的遞送 ID。
- 在實際發送前後記錄檢查點。
- 重啟時，等幂地重放擱置中的區塊並避免重複。

### 5) 遷移與切換

- 第 1 階段：影子模式（新管線計算輸出但由舊路徑發送；進行比較）。
- 第 2 階段：按運行時逐一切換（`acp`，然後 `subagent`，接著 `main` 或按風險反向順序）。
- 第 3 階段：刪除舊版特定於運行時的串流程式碼。

## 非目標

- 此次重構不會改變 ACP 原則/權限模型。
- 除了投射相容性修復外，不進行特定於通道的功能擴展。
- 不重新設計傳輸/後端（acpx 外掛程式合約保持原樣，除非需要實現事件對等性）。

## 風險與緩解措施

- 風險：現有 main/subagent 路徑的行為回歸。
  緩解措施：影子模式差異比對 + 介面卡合約測試 + 通道端到端測試。
- 風險：崩潰恢復期間重複發送。
  緩解措施：持久的遞送 ID + 遞送介面卡中的等幂重放。
- 風險：運行時介面卡再次分岐。
  緩解措施：所有介面卡必須通過共用的合約測試套件。

## 驗收標準

- 所有運行時均通過共用的串流合約測試。
- Discord ACP/main/subagent 對於微小差異產生等效的間距/分塊行為。
- 崩潰/重啟重放不會為相同的遞送 ID 發送重複的區塊。
- 舊版 ACP 投射器/合併器路徑已移除。
- 串流配置解析是共用的，並且與運行時無關。
